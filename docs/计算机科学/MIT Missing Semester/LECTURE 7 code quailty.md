### 代码质量工具与评判标准概览

| 评判领域 | 工具/技术类型 | 核心关注点 (评判标准) | 典型工具示例 | 实用建议 |
| :--- | :--- | :--- | :--- | :--- |
| **格式化 (Formatting)** | 自动格式化器 | **表层语法一致性**：缩进、空格、引号、`import` 顺序、行宽。 | Prettier (多语言), Black (Python), gofmt (Go) | 将配置文件存入版本控制；配置 IDE “保存时自动格式化”。 |
| **Lint (静态分析)** | Linter | **代码逻辑与反模式**：检测潜在 Bug、不必要的嵌套、危险函数调用、代码异味。 | Ruff (Python), ESLint (JS), semgrep (语义检查) | 优秀的 Linter 会解释“为什么这条规则不好”并给出替代方案。 |
| **正确性 (Testing)** | 软件测试 | **行为符合预期**：函数输出、模块交互、端到端流程、回归测试、性质测试。 | pytest (Python), Hypothesis (基于性质) | 对外部依赖（数据库/API）使用 Mock；发现 Bug 后必写回归测试。 |
| **覆盖率 (Coverage)** | 代码覆盖率工具 | **测试完备性**：统计哪些代码行在测试中被执行到，识别未覆盖的逻辑路径。 | Coverage.py, Codecov (Web 服务) | 不要盲目追求 100% 覆盖率，高质量的测试用例比数字更重要。 |
| **提交关卡 (Pre-commit)** | Git 钩子 | **代码入库质量**：在提交代码前强制运行格式化、Lint 和基础测试。 | pre-commit 框架 | 确保不符合标准的代码“无法提交”，从源头保证代码库整洁。 |
| **自动化集成 (CI)** | 持续集成服务 | **环境兼容性与长期维护**：在不同 OS 和版本上验证代码，确保主分支始终可编译/通过。 | GitHub Actions | 在 PR 上运行 CI 以发现贡献者的错误；通过矩阵测试验证多环境兼容性。 |
| **规范化部署 (CD)** | 持续部署 | **发布一致性**：自动化构建二进制文件、Docker 镜像或部署静态网站。 | GitHub Pages (静态), Docker | 确保代码 push 后能自动、可靠地更新到生产环境。 |
| **命令运行 (Runners)** | 命令运行器 | **操作复杂性**：统一项目内各种工具的调用方式，降低开发者的认知负担。 | just, npm scripts, Hatch | 用 `just lint` 代替又长又难记的命令行参数。 |

### 特别补充：正则表达式 (Regex) 评判标准
在该课程中，正则表达式被视为一种**模式匹配的标准**，其评判标准在于：
* **精确度**：是否能准确匹配目标字符串（如日期、邮箱），同时避免误报。
* **贪婪性控制**：在解析复杂结构（如 JSON）时，是否正确使用了非贪婪模式。
* **局限性认知**：是否意识到 Regex 无法处理非正则语言（如嵌套的 HTML）。
